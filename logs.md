### 19. 식별자 명명 규칙과 기호의 시각적/의미론적 이해

#### 식별자 대소문자 구분 (PascalCase vs snake_case)
- **PascalCase (대문자 시작)**: 타입(Type), 열거형(Enum), 구조체(Struct), 열거형 변종(Variant) 이름에 사용. (예: `Command`, `Command::Uppercase`)
- **snake_case (소문자 시작)**: 변수(Variable), 함수(Function), 메서드 이름에 사용. 
- **활용 예시**: `for (string, command) in input`에서 구조 분해 할당으로 꺼낸 튜플의 요소들은 실제 데이터를 담는 변수이므로 소문자 `command`를 사용.

#### -> (Thin Arrow) vs => (Fat Arrow)
- **`->` (Thin Arrow)**: "어떤 **타입**의 결과물을 내놓을 것인가?"
    - **연상 팁**: **T**hin = **T**ype
    - **비유**: 식당 메뉴판 가격표처럼 함수 선언부 등에서 반환 타입을 명시.
- **`=>` (Fat Arrow)**: "이 상황에는 이 **흐름(행동)**을 실행해라"
    - **연상 팁**: **F**at = **F**low
    - **비유**: 리모컨의 버튼처럼 `match` 문의 각 팔(Arm)에서 특정 패턴과 실행할 코드를 연결. (Javascript의 화살표 함수와 다름)
    - **추가 용도**: `macro_rules!`에서 매크로 입력 패턴과 확장 코드를 연결할 때 사용.

### 20. Option 타입과 패턴 매칭 심화 (if let, while let)

#### Option, Some, None의 관계 (타입과 값)
- **`Option`**: "타입(Type)"의 이름으로 설계도 혹은 진열대의 안내문구 역할. 값이 없을 수도 있는 상태를 강제하여 `null`로 인한 런타임 에러(NullPointerException 등)를 미연에 방지함. (예: `Option<u16>`)
- **`Some` / `None`**: `Option` 타입 안에 속하는 실제 데이터(Variant/Value). `Option`이 상자라는 '분류'라면, `Some(5)`은 진짜 내용물이 든 상자, `None`은 텅 빈 상자를 의미함.

#### Option 상자 열기 비교
1. **`.unwrap()`**: "이 안에 무조건 값이 있을 거야! 당장 까줘!" (값이 없으면 Panic 발생)
2. **`match`**: 모든 경우의 수(`Some`과 `None`)를 다 검사. (가장 안전하지만 길다)
3. **`if let`**: "값이 있으면 까서 쓰고, 아니면 쿨하게 버리자!"
    - 문법: `if let [기대하는 패턴] = [검사할 데이터] { ... }`
4. **`while let`**: 값이 들어있는 동안 반복해서 꺼내서 쓴다. (조건부 루프)

#### 중첩 패턴 매칭 (Nested Option)
- 리스트(Vec)의 내용물이 `Some`으로 포장되어 있을 때, 이를 `.pop()`으로 꺼내면 겹상자가 됨. (예: `Some(Some(숫자))`)
- 이때 `while let Some(Some(integer)) = vec.pop() { ... }` 처럼 왼쪽 패턴 모양을 거울처럼 맞춰주어 두 겹 포장을 한 번에 벗길 수 있음.

#### 관용적인 변수명: `cursor`
- 데이터 혹은 배열 등을 순서대로 읽거나 위치를 검증할 때 가리키는 포인터/인덱스 역할의 변수를 프로그래밍에서 관례적으로 `cursor`라고 지칭함. (이 코드는 역순으로 -1씩 움직임)

### 21. 패턴 매칭에서 `&` 와 `ref` 의 차이

#### `match` 구문 (값을 받는 패턴 자리)에서의 의미
- **`&p` (역참조/해체)**: "이 상자 안에는 이미 참조(주소)가 들어있으니, 그 참조 껍데기를 벗겨내고 **실제 값**을 `p`에 바인딩해라."
    - 에러 발생 상황: 상자 안에 참조가 아니라 실제 데이터(`Point` 구조체 등)가 들어있을 때 껍데기를 벗기려 하면 타입 불일치 에러가 발생함.
- **`ref p` (새 참조 생성)**: "이 상자 안의 실제 값을 밖으로 꺼내서 소유권을 빼앗지 말고, 그 값이 있는 곳의 **새로운 참조(주소)**만 만들어서 `p`에 바인딩해라."
    - 해결 상황: 실제 데이터를 소유권 이동(Move) 없이 빌려와야 할 때 사용. `Some(ref p)`를 쓰면 `p`는 `&Point` 타입이 되어 원본 변수가 안전하게 보존됨.

#### 데이터 타입에 따른 소유권 이동(Move) vs 복사(Copy)
- **복사 가능한 타입 (`i32`, `bool` 등)**: 값 자체가 가볍고 `Copy` 트레이트가 구현되어 있어, 참조에서 값을 꺼내와도 소유권 이동(Move)이 아니라 복사(Copy)가 일어납니다. 원본 데이터는 무사합니다.
- **복사 불가능한 타입 (`String`, 일반 구조체 등)**: 빌려온 상태(참조)에서 강제로 실제 값을 꺼내어 소유권을 가져오려 시도하면, 남의 물건을 훔치는 꼴이 되므로 컴파일러가 소유권 규칙 위반 에러를 냅니다.

### 18. 메모리 구조와 가변성(Mutability)의 본질

#### 가변 참조(&mut) vs 소유권(Ownership)의 메모리 구조
- **공통점**: 실제 데이터 내용(예: "테슬라")은 **힙(Heap)**에 저장됨.
- **차이점**:
    - **소유권자**: 스택에 [주소, 길이, 용량] 정보를 담은 '등본'을 가짐. 물건의 생사여탈권을 쥐고 있음.
    - **가변 참조자**: 스택에 '소유권자의 주소'를 가리키는 포인터를 가짐. 잠시 물건을 독점적으로 고칠 수 있는 '수리 열쇠'를 빌린 상태.
- **Move의 효율성**: 소유권이 이동할 때 힙의 데이터를 복사하는 것이 아니라, 스택에 있는 작은 정보(등본)만 넘겨주기 때문에 매우 빠름.

#### mut 키워드의 의미
- `let mut`은 해당 변수가 점유한 메모리 공간이 수정 가능함을 선언하는 **'허가증'**.
- 이 허가증이 있어야만 데이터의 크기를 변경하거나(Heap 재할당 가능성), 타인에게 가변 참조(&mut)를 빌려줄 자격이 생김.

#### Deep Dive: 스택(Stack)과 로우 레벨(Low-level)
- **스택의 특성**: CPU의 전용 레지스터(Stack Pointer)와 명령(PUSH/POP)으로 직접 관리됨. 하드웨어와 가장 가깝고 단순한 '로우 레벨' 영역.
- **힙의 특성**: 운영체제나 소프트웨어 할당자가 관리하는 추상화된 영역. 유연하지만 관리가 복잡함.

### 17. 해시맵(HashMaps) 재도전: 빌림 검사기와 소유권의 실전

#### 빌림 검사기(Borrow Checker)와 NLL
- **문제 발생**: 하나의 HashMap(`scores`)에서 동시에 두 개의 가변 참조(`team_1`, `team_2`)를 만들려고 할 때 발생.
- **NLL (Non-Lexical Lifetimes)**: 현대 Rust에서는 변수가 더 이상 사용되지 않는 시점을 컴파일러가 똑똑하게 파악하여 빌림(Borrow)을 조기에 종료시킴.
- **해결책**:
    1. **명시적 스코프**: `{ }` 중괄호를 사용해 빌림의 범위를 강제로 제한.
    2. **순차적 처리**: 첫 번째 가변 참조 사용이 완전히 끝난 뒤에 두 번째 참조를 생성.

#### Entry API의 실전 활용
- **핵심**: `entry(key).or_default()`는 데이터를 삽입하거나 가져온 뒤, 즉시 수정할 수 있는 **가변 참조자(&mut T)**를 반환함.
- **효율성**: 별도의 `insert`를 호출할 필요 없이, 반환된 참조를 통해 필드 값을 직접 수정하면 메모리 안의 데이터가 즉시 업데이트됨.

#### 흔한 실수: 오타와 변수명
- 파싱 단계에서 생성한 변수명(`team_2_score`)과 실제 업데이트 시 사용한 이름(`team_2_goals`)이 다를 경우 컴파일 에러가 발생하므로 주의가 필요함.

#### Deep Dive: 가변 참조자(&mut)와 인플레이스(In-place) 업데이트
- **핵심 원리**: `team_1`은 해시맵 외부로 데이터를 꺼내온 것이 아니라, 해시맵 **내부 메모리 공간**을 가리키는 주소(가변 참조자)를 들고 있음.
- **재할당 불필요**: 이 주소를 따라가서 값을 직접 수정하면 해시맵 원본이 즉시 변경되므로, 별도의 `insert`나 재할당 과정 없이도 함수 마지막에 업데이트된 해시맵을 반환할 수 있음.
- **비유**: 지도를 새로 그리는 것이 아니라, 기존 지도의 특정 위치를 찾아가서 그 자리에서 내용을 고치는 것과 같음.

### 16. Default 트레이트의 원리와 Entry API 심화

#### Default 트레이트: "자동 생성 자격증"
- **개념**: 타입이 "표준적인 기본값"을 어떻게 만드는지 정의하는 자격증(Trait).
- **#[derive(Default)]**: 구조체 위에 붙여 컴파일러가 자동으로 기본값 생성 로직(숫자 0, 빈 문자열 등)을 작성하게 함.
- **필수성**: `or_default()`를 사용하려면 해당 타입이 반드시 `Default` 트레이트를 구현하고 있어야 함.

#### or_default() vs or_insert()
- **or_default()**: "자격증(Default)에 적힌 대로 표준적인 시작값을 만들어라." (보통 0점 시작)
- **or_insert(값)**: "자격증 유무와 상관없이 내가 지금 건네주는 이 특정 값을 넣어라." (커스텀 값 시작)

#### Entry API의 작동 단계
1. **entry(key)**: 해당 키의 보관함 위치(권한)를 선점.
2. **or_...()**: 자리가 비었을 때만 데이터를 채우고, 최종적으로 그 값에 대한 **가변 참조자(&mut T)**를 반환.
3. **수정**: 반환된 주소를 통해 한 줄로 즉시 데이터 수정 가능.

### 15. 해시맵(HashMaps) 심화: 구조체 연동과 지연 평가

#### HashMap의 중첩 구조 (Key -> Struct)
- **개념**: 해시맵의 값(Value)으로 단순 숫자가 아닌, 여러 필드를 가진 **구조체(Struct)**를 저장할 수 있음.
- **예시**: `HashMap<&str, TeamScores>` 구조에서 팀 이름을 키로, 득점/실점 정보가 담긴 구조체를 값으로 관리.
- **장점**: 연관된 여러 데이터를 하나의 키 아래에 응집력 있게 보관 가능.

#### Iterator와 지연 평가 (Lazy Evaluation)
- **동작 원리**: `.split()`이나 `.lines()` 등을 호출했을 때 즉시 데이터를 쪼개서 메모리에 담지 않음.
- **특징**: 반복자(Iterator)는 "어떻게 일할지"에 대한 계획과 "현재 위치" 정보만 가진 상태로 대기함.
- **장점**: `next()`를 호출할 때만 실제로 계산을 수행하므로, 필요한 만큼만 일하게 되어 메모리와 CPU 자원을 극도로 아낄 수 있음 (Zero-cost abstraction).

#### Entry API와 `or_default()`
- **로직**: `scores.entry(team_name).or_default()`
- **작동**: 
    1. 해당 키가 있는지 확인.
    2. 있다면 기존 값에 대한 가변 참조를 반환.
    3. 없다면 `#[derive(Default)]`가 구현된 구조체의 기본값(0, 0 등)을 즉시 생성하여 삽입 후 반환.
- **효과**: "있으면 가져오고 없으면 새로 만들어라"라는 복잡한 조건문을 단 한 줄의 선언적인 코드로 해결.

#### 메서드 체이닝과 unwrap
- **흐름**: `next().unwrap().parse().unwrap()`
- **의미**: 
    - `next()`: 반복자에서 조각 하나 꺼내기.
    - `parse()`: 꺼낸 문자열을 타겟 타입(예: `u8`)으로 변환.
    - `unwrap()`: 성공했다고 믿고 상자를 열어 알맹이만 꺼내기 (실패 시 패닉).
- **타입 추론**: `parse()`는 변수에 지정된 타입(예: `: u8`)을 보고 무엇으로 변환할지 스스로 결정함.

### 14. 해시맵(HashMaps)과 Rust 테스트 시스템의 가시성

#### HashMap<K, V>: "이름표가 붙은 보관함"
- **특징**: 키(Key)와 값(Value)의 쌍을 저장하는 컬렉션. Python의 딕셔너리나 JS의 Map과 유사함.
- **선언**: `use std::collections::HashMap;`을 명시적으로 가져와야 사용 가능.
- **조작**:
    - `HashMap::new()`: 빈 해시맵 생성.
    - `.insert(key, value)`: 데이터를 넣거나 기존 값을 업데이트.
- **제약**: 모든 키는 같은 타입이어야 하고, 모든 값도 같은 타입이어야 함.

#### Rust 테스트 시스템의 출력 관리 (println!)
- **현상**: 테스트 코드 안에 `println!`을 작성해도 테스트가 성공하면 터미널에 출력되지 않음.
- **이유**: Rust 테스트 러너는 성공한 테스트의 표준 출력(stdout)을 기본적으로 캡처(숨김)하여 결과 화면을 깨끗하게 유지함.
- **확인 방법**:
    1. **의도적 실패**: `assert!(false)` 등을 통해 테스트를 실패시키면, 그때까지의 출력이 모두 표시됨(디버깅 용도).

### 10. 모듈(Modules)과 가시성(Visibility)

#### 기본 원칙: "닫힌 문이 기본"
- Rust의 모든 항목(함수, 구조체, 상수 등)은 선언된 모듈 내부에서만 볼 수 있는 **비공개(private)** 상태가 기본임.

- **방법**: `pub use self::inner::item as simple_name;`
- **효과**: 내부 모듈 구조를 숨기면서 필요한 기능만 콕 집어서 외부에 공개할 수 있음(캡슐화).

- **별칭(Alias)**: `as` 키워드로 충돌 방지나 가독성을 위해 이름을 새로 붙일 수 있음.

- `duration_since` 처럼 결과가 실패할 수 있는 작업은 `Result`를 반환하며, `match` 문을 통해 `Ok`와 `Err`를 명확히 구분하여 처리하는 것이 Rust의 표준적인 안전 장치임.

### 11. 구문 기호의 시각적 의미

#### `->` (가는 화살표): "결과물은 이것이다"
- 용도: 함수의 반환 타입(Return Type)을 지정할 때 사용.
- 의미: 데이터가 함수의 입력에서 출력 방향으로 흘러가는 '통로'를 상징함.
- 비유: "이 기차의 목적지(타입)는 어디인가?"

#### `=>` (굵은 화살표): "그렇다면 이것을 해라"
- 용도: `match` 문의 각 케이스(Arm)와 실행 코드를 연결할 때 사용.
- 의미: 조건이 참일 때 특정 행동으로 이어지는 '강한 논리적 결합'을 상징함. (수학의 임플라이 기호와 유사)
- 비유: "이 갈림길을 선택했다면 어디(실행)로 가야 하는가?"

#### 기호가 다른 이유
- 정적인 정보(타입 정의)와 동적인 흐름(코드 실행)을 시각적으로 즉시 분리하여 가독성을 높이기 위함.

### 11. 문자열(Strings)의 두 얼굴

#### `&str` (문자열 슬라이스): "비석에 새겨진 각인"
- **특징**: 프로그램 바이너리(실행 파일)에 고정된 데이터. 불변이며 메모리 할당 비용이 없음(매우 빠름).
- **기본 포지션**: Rust에서 `"..."` 처럼 따옴표로 쓴 리터럴은 기본적으로 `&str` 타입임.
- **성능 중심**: 굳이 수정할 필요 없는 고정 텍스트를 가장 가볍게 처리하기 위한 Rust의 설계 선택.
- **왜 '슬라이스(Slice)'인가?**:
    - **부분의 의미**: 전체 데이터 덩어리(String이나 바이너리 데이터 섹션)에서 원하는 특정 구간만 '조각(Slice)' 내어 가리킬 수 있기 때문.
    - **내부 구조**: [시작 주소(Pointer) + 길이(Length)]라는 메타데이터만 가진 Fat Pointer임. 즉, 데이터 전체를 복사하는 게 아니라 "어디서부터 얼마나 볼 것인가"라는 범위 정보만 가짐.
    - **리터럴의 정체**: 코드에 쓴 `"Hello"`도 실행 파일의 거대한 데이터 뭉치 속에서 특정 주소와 길이를 잘라낸(Slice) 조각임.

#### `String` (소유된 문자열): "이동식 칠판"
- **특징**: 실행 중에 힙(Heap) 메모리에 새로 할당된 데이터. 크기를 늘리거나 내용을 수정할 수 있음.
- **소유권**: 데이터를 통째로 들고 다니는 소유권자이므로, 함수에 넘겨줄 때 주의가 필요함.

#### 상호 변환 규칙
1.  **`&str` -> `String`**: `.to_string()` 또는 `String::from()` 사용. 비석의 글자를 복사해 새 칠판에 옮겨 적는 행위. (메모리 할당 발생)
2.  **`String` -> `&str`**: `&`를 붙여 대여(Borrow)함. 칠판 내용을 잠깐 보여주기만 하는 행위. (추가 메모리 할당 없음)

#### 실무적 팁: 함수 인자 설계
- 함수 인자로 `&str`을 받으면, `&str` 리터럴은 물론 `&String`도 전달받을 수 있어 유연성이 극대화됨.
- 반면 `String`을 인자로 받으면 메모리 소유권을 강제로 뺏어오게 되므로, 꼭 필요한 상황이 아니라면 `&str`을 받는 것이 권장됨.

### 12. 함수 반환 타입의 결정 기준 (strings3 실전)

#### `&str`을 반환할 때: "범위만 새로 지정"
- **예시**: `trim()` 함수.
- **원리**: 원본 데이터를 수정하지 않고, 공백을 제외한 '진짜 내용'이 시작되는 주소와 길이 정보만 새로 계산해서 반환함.
- **특징**: 메모리 추가 할당 없이 원본이라는 도화지 위에서 시선(포인터)만 옮기는 것이라 매우 효율적임.

#### `String`을 반환할 때: "새로운 데이터 생성"
- **예시**: `format!`, `replace`, `to_string() + "..."` 등.
- **원리**: 원본에 없는 새로운 글자를 덧붙이거나 내용을 치환하면, 이 결과물을 담을 '새 집(Heap 메모리)'이 반드시 필요함.
- **안전성 (댕글링 포인터 방지)**: 함수 안에서 만든 임시 데이터는 함수가 끝나면 사라짐. 따라서 사라질 데이터의 주소(`&str`)를 내보내는 대신, 데이터 자체를 `String` 상자에 담아 호출자에게 **소유권**을 통째로 넘겨줘야 안전하게 전달됨.

#### 문자열 더하기 (`+`) 연산자의 규칙
- `input.to_string() + " world!"` 처럼 사용 가능.
- **조건**: 왼쪽 항은 반드시 소유권이 있는 `String`이어야 하고, 오른쪽 항은 참조 타입인 `&str`이어야 함.
- **이유**: 내부적으로 `String`의 `add` 메서드가 작동하여 왼쪽 String에 오른쪽 내용을 이어 붙이는 방식이기 때문.

### 13. 소유권 있는 문자열 생성과 타입 추론 (strings4 실전)

#### `String::from()` vs `to_string()`
- **결론**: `&str`을 `String`으로 만들 때는 기능적으로 동일함.
- **차이**: `String::from`은 `From` 트레이트를, `to_string`은 `ToString` 트레이트를 사용함. 보통 "생성"의 의미가 강한 `String::from`이 자주 선언됨.

#### `to_owned()`
- **의미**: "빌린 데이터(&str)를 내 소유의 데이터(String)로 만들겠다"는 명시적 요청. `&str`을 복사하여 새로운 힙 메모리에 `String`을 생성함.

#### `into()` 의 마법 (Type Inference)
- **원리**: "내가 가진 데이터를 호출된 곳이 원하는 타입으로 변신시키겠다"는 뜻.
- **작동**: 함수가 `String`을 원하면 `String`으로, `&str`을 원하면 `&str`로 컴파일러가 문맥을 보고 자동으로 변환해줌.

#### `to_lowercase()` 가 `String`을 반환하는 이유
- **가변성**: 대문자가 소문자로 바뀔 때 유니코드상 **바이트 크기**가 달라질 수 있음.
- **불가피성**: 원본 메모리(비석)는 크기가 고정되어 있어 크기가 변할 수 있는 데이터를 덮어쓸 수 없음. 따라서 무조건 새로운 메모리(String)를 할당해야 함.

#### `&[0..1]` 슬라이싱의 결과
- `String`을 생성했더라도, 그 뒤에 `[0..1]` 범위를 지정하고 앞에 `&`를 붙이면 다시 `&str`이 됨. 이미 생성된 상자의 '일부 범위'를 가리키는 참조자가 되기 때문.

### 9. 열거형(Enums) 설계와 안전성 실전

#### 열거형의 근본적인 목적: "단일 통로 관리"
- 흩어진 여러 함수(process_a, process_b...)를 호출하는 대신, 모든 명령을 하나의 열거형(Message) 봉투에 담아 하나의 함수(process)로 집중시킴.
- 중앙 집권식 관리를 통해 "할 일 목록(Queue)"을 만들거나 시스템의 상태를 일관되게 통제할 수 있음.

#### Rust vs 자바스크립트(JSON) 데이터 접근 철학
- **JS (희망 기반 접근)**: 데이터가 있을 거라 믿고 `msg.move.x`와 같이 접근. 데이터가 없으면 `undefined`를 반환하고, 실제 사용 시점에서 런타임 에러 발생.
- **Rust (확신 기반 접근)**: "신분 확인(match)" 없이는 데이터에 손도 못 대게 설계됨.
- **안전성**: match 문은 모든 케이스를 다루도록 강제(Exhaustiveness)하므로, 데이터 구조가 바뀌거나 타입이 누락되었을 때 컴파일 단계에서 즉시 발견 가능.

#### 동작 메커니즘: &mut self와 주인공(Instance)
- **주인공 생성**: `let mut state = State { ... };` 메모리에 실제 데이터 판이 깔림.
- **명령 하달**: `state.process(Message::...)` 호출 시, `state`는 `self`라는 이름으로 함수에 대여됨.
- **수행**: `&mut self`는 원본을 가리키는 '긴 장갑'과 같아서, 함수 내부에서 `self.width = ...`를 수행하면 멀리 있는 원본 `state`의 값이 직접 수정됨.

#### 구조체(AND) vs 열거형(OR)
- **구조체**: 데이터의 **구성 요소**를 정의. (A 와 B 와 C가 모두 실재함)
- **열거형**: 데이터의 **정체/상태**를 정의. (A 혹은 B 혹은 C 중 딱 하나만 실재함)
- **결합**: 서로 다른 모양의 구조체들을 열거형이라는 큰 봉투에 담아 전달함으로써, 타입의 통일성과 데이터의 독립성을 동시에 확보(Sum Type).

#### 시스템 설계의 0단계: 설계도
- 런타임에 즉흥적으로 데이터를 생성하는 것이 아니라, 구조체(데이터)와 열거형(메시지 체계), impl(능력)을 사전에 엄격히 정의해야 함.
- 이 엄격함이 "실행 중에 데이터 구조 문제로 프로그램이 죽는 일"을 원천 차단함.

#### 구문적 특징: 중괄호`{}` vs 소괄호`()`
- **Enum/Struct 세계 (`{}`)**: 데이터의 모양이나 상자를 설계/생성/해체할 때 사용. 설계도에 중괄호를 썼다면 사용할 때도 중괄호를 써서 필드명을 맞춰야 함.
- **함수/메서드 세계 (`()`)**: 기계나 동작을 가동할 때 사용. 인자를 전달하는 '행위'를 의미함.
- **연결**: `match` 문에서 Enum 상자를 `{}`로 뜯어 알맹이를 꺼낸 뒤, 그 알맹이를 함수의 `()`에 집어넣는 식으로 두 세계가 연결됨.

#### 명령형(Imperative) vs 선언형(Declarative) 명령
- **직설적 명령 (동사)**: `move_hero(x, y)` - 즉시 실행되는 행위. 로그 기록이나 취소가 어렵고 실행 시점에 종속됨.
- **데이터화된 명령 (명사)**: `Command::Move { x, y }` - 행위의 의도를 데이터로 캡슐화한 것.
- **장점**: 명령을 변수에 담아 보관(로그), 멀리 전송(네트워크), 나중에 실행(지연), 실행 취소(Undo) 등이 가능해져 시스템의 유연성이 극대화됨.

### 6. 구조체 (Structs)

#### 구조체의 세 가지 형태
- 일반 구조체 (Regular Struct): 관례적인 필드 이름을 가짐. `struct Color { red: u32, ... }`
- 튜플 구조체 (Tuple Struct): 필드 이름 없이 순서(인덱스)로 접근. `struct Color(u32, u32, u32);`
- 유닛 구조체 (Unit Struct): 필드가 전혀 없는 구조체. 데이터 저장보다는 타입 그 자체의 존재나 특정 능력을 부여할 때 사용. `struct Unit;`

#### 구조체 업데이트 구문 (..)
- 기존 구조체 인스턴스의 값을 바탕으로 새로운 인스턴스를 만들 때 사용.
- 바꿀 필드만 명시하고 나머지는 `..원본` 문법으로 일괄 처리 가능.
- 주의: 소유권이 있는 타입(String 등)을 명시적으로 적지 않고 `..`에 맡기면 소유권이 새 인스턴스로 이동됨.

#### derive(Debug)와 {:?}
- 구조체는 기본적으로 출력 기능이 없음.
- `#[derive(Debug)]`를 구조체 위에 선언하면 컴파일러가 출력용 코드를 자동으로 생성함.
- `println!("{:?}", 변수)`처럼 `{:?}` 포맷과 함께 사용하여 내부 값을 확인 가능.

### 7. 소유권과 타입 특성

#### Copy 트레이트와 값 복사
- 숫자(i32, u32 등), 불리언(bool), 문자(char) 등 크기가 고정된 가벼운 타입은 소유권이 이동하지 않고 **복사**됨.
- 따라서 숫자를 함수 인자로 넘겨도 호출한 쪽에서 원본을 계속 사용할 수 있음.

#### 함수의 반환 법칙
- 함수 내부 마지막 줄에 세미콜론(`;`)이 없으면 해당 줄의 결과값이 함수 밖으로 반환됨.
- 세미콜론이 있으면 값이 소멸되고 `()` (Unit) 타입을 반환하게 됨.

#### 에러 코드 E0308
- 타입 불일치(Mismatched Types) 상황에서 발생.
- 컴파일러가 기대한 타입(expected)과 실제 발견된 타입(found)을 비교하여 알려줌.

### 8. 열거형 (Enums)

#### 열거형의 정의와 특징
- 관련된 여러 선택지(변체, Variants)를 하나의 타입으로 묶음.
- 각 변체는 서로 다른 형태의 데이터를 담을 수 있음 (단순 이름, 튜플 형태, 구조체 형태).

#### 데이터 활용 방식
- 재사용 방식 (Recommended): 외부의 구조체를 타입으로 사용. `Move(Point)`
- 인라인 방식: 열거형 내부에 직접 필드를 정의. `Resize { width: i32, height: i32 }`
- 장점: 익명 데이터를 즉석에서 정의하는 것보다 이미 정의된 타입(Point 등)을 재활용하는 것이 기능 확장성 면에서 뛰어남.

#### 열거형의 impl 구현
- 구조체와 마찬가지로 열거형에도 `impl` 블록을 통해 메서드를 추가할 수 있음.
- 데이터(Enum)와 그 데이터를 다루는 로직(impl)을 분리하여 응집력 있는 설계가 가능함.
- 예: DB에서 가져온 상태(Enum)에 따라 사용자 메시지를 동적으로 생성하는 기능을 impl에 구현.

#### 안전한 데이터 접근
- JSON처럼 `message.move.x`와 같이 직접 접근하는 것은 불가능함.
- 현재 열거형이 어떤 상태인지 모르기 때문에, 반드시 패턴 매칭(match 등)을 통해 안전하게 데이터를 꺼내 써야 함.

### 4. 소유권 (Ownership) & 이동 시맨틱 (Move Semantics)

#### 이동 (Move)
- 변수를 다른 변수에 대입하거나 함수 인자로 넘기면 소유권(집문서)이 통째로 넘어간다.
- 넘겨준 원본 변수는 더 이상 쓸 수 없다. (Shadowing이나 Copy 트레이트가 없는 한)

#### 참조/대여 (Borrowing)
- &를 붙여 주소표만 건네주는 행위. 원본 소유권은 그대로 유지된다.
- 불변 참조(&): 여러 명이 동시에 구경은 가능하지만 수정은 불가.
- 가변 참조(&mut): 딱 한 명만 빌려서 수정할 수 있음.

#### 수정(Push)과 메모리 주소
- Vec이나 String에 데이터를 push할 때, 메모리 여유 공간이 있다면 주소는 유지된 채 내용만 업데이트된다.
- 소유권이 이동(Move)한다고 메모리 주소가 매번 바뀌는 것이 아니라, 집문서 명의자만 바뀌고 집(메모리 주소)은 그대로인 경우가 많아 효율적이다.

#### 인자로 전달 시 매개변수에서 소유권 이동과 동시에 가변성 확보
- 함수의 매개변수에 `mut vec: Vec<i32>`와 같이 작성.

#### 가변 참조자(&mut)와 대여 규칙(Borrowing Rules)
- &mut x: 원본을 파괴하지 않고 주소를 찾아가서 내용물을 수정할 수 있는 독점적인 권한을 빌려오는 것.
- 독점권: 어떤 값에 대해 가변 참조자(&mut)가 활성화되어 있다면, 그 시점에는 오직 그 한 명만 값을 보거나 고칠 수 있다.
- 제약 사항: 동일한 값에 대해 두 개 이상의 가변 참조자를 동시에 가질 수 없다. (동시에 물건을 고치려 할 때 발생하는 데이터 충돌 방지)
- 해결법: 가변 참조자의 사용 범위를 겹치지 않게 순서를 조정하거나, 사용이 끝난 뒤 다음 참조를 선언해야 한다.

#### 소유권 대여와 데이터 재사용 (move_semantics5)
- 조회 작업 시 &를 사용하여 대여(Borrow)하면, 함수 종료 후에도 소유권이 유지되어 원본 데이터를 다른 함수(예: 소유권을 가져가는 변환 함수)에서 다시 사용할 수 있다.
- 함수 선언에 반환 타입(-> T)을 명시했다면, 바디 마지막에 해당 타입의 값을 반드시 반환해야 한다. 단순히 출력만 하고 결과를 외부에서 쓰지 않는다면 반환 타입을 제거해도 무방하다.
- println! 매크로 사용 시 주의: `println!("{}", value)`와 같이 포맷 문자열을 반드시 첫 번째 인자로 사용해야 한다.

### 2. 타입 시스템 심화 (String vs &str)

#### &str (String Slice - 비석에 새긴 각인)
- 프로그램 바이너리에 박혀 있음. 메모리를 따로 빌릴 필요 없이 주소만 알면 되기에 빠름.
- 읽기 전용이며 크기를 키우거나 내용을 고칠 수 없음.

#### String (Owned String - 이동식 칠판)
- 실행 중 힙(Heap) 메모리에 할당됨. 내용을 수정하거나 크기를 늘릴 수 있음.
- 소유권(집문서)이 명확히 존재하며, 주인을 따라다니며 책임짐.

#### 실무 팁
- 데이터를 생성/저장하거나 DB에서 가져올 때는 String을 사용.
- 함수 인자로 읽기 전용 데이터를 넘길 때는 &str을 사용하는 것이 국룰 (유연성 및 효율성).

### 5. 유용한 도구들

#### 반복자 (Iterator)
- `input.iter().map(|x| x + 1).collect()`
- iter(): 데이터를 하나씩 꺼낼 준비 (도우미 호출)
- map(): 꺼낼 때마다 어떻게 가공할지 명령 (게으른 연산: 실제론 이때 안 일어남)
- collect(): 드디어 실제로 가공해서 새 바구니에 담기 (실행 시점, 반환 타입 기반으로 그릇 결정)

### 3. 매크로 (Macro) vs 함수 (Function)

#### 매크로의 정체 (!)
- 컴파일 시점의 마법: 컴파일러가 실행 파일이 만들어지기 전, 짧은 코드(println!)를 사전에 정의된 복잡한 코드로 확장(Expand)해줌.
- 사용 이유: 가변 인자 처리, 코드 자동 생성, 컴파일 타임의 세밀한 제어.
- 명시성: 이름 뒤에 !를 붙여 일반 함수와 시각적으로 구분함.
